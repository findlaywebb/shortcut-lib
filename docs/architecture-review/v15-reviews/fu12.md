## Per-validator assessment

### 1. `envelope-mismatch` — PASS

The oracle is loaded correctly via `_load_oracle()` which reads `data/observed_envelope_types.json` (path computed relative to `__file__`). The JSON shape interpretation is correct: the code descends `oracle["slots"][action_id][slot_key]["envelopes"]` (lines 212–217), and the oracle top-level keys are confirmed as `["actions_observed", "bare_string_slots", "generated_at", "samples_scanned", "slots"]` — matching the `oracle.get("slots", {})` access pattern. The test (`test_validate_envelope_mismatch_flagged`) exercises the live oracle against `is.workflow.actions.downloadurl` / `WFURL`, which the oracle correctly records as `WFTextTokenString`-only — the mismatch is caught and the message includes both the wrong type and the expected type. The `unknown-envelope` info-severity path (novel action + novel type → `WFTotallyInventedType12345`) is also tested and passes.

One performance note: the `unknown-envelope` branch recomputes `all_observed_types` by iterating the entire oracle on every novel-slot miss (line 225). With a 195-action, 214-slot oracle this is negligible, but it could be cached at call-time for free. Low priority for V1.5.

### 2. `variable-not-set` — YELLOW (false-negative gap, documented)

The validator pre-collects all variable names bound anywhere in the flat action list (`_collect_set_variable_names`, line 246) before checking references. This means a use-before-set pattern — where action 0 references `MyVar` and action 1 is the `SetVariable` that binds it — is **not flagged**. The brief asked for sequential walk ("build the name set as it goes") but the implementation chose global pre-collection.

The implementation's docstring is transparent about this: "A name set inside a `Repeat` body is visible after the loop" and "the check is conservative (no false negatives)." That framing is defensible for a V1 conservative validator — iOS Shortcuts does propagate variable names across control-flow boundaries — but it equally suppresses out-of-order references in flat linear flows, which are genuine errors. No test covers the use-before-set case, so the gap is unverified.

Magic-var handling is structurally sound: `MagicVar.to_token()` returns `{"Type": "CurrentDate"}` (no `VariableName` key), so magic-var tokens are filtered before the `VariableName` check ever runs. The `_MAGIC_VAR_TYPES` frozenset is a secondary guard for the edge case where a user names a variable after a magic-var type string (e.g. `NamedVar("CurrentDate")`). The set matches `values.py` singletons exactly (confirmed: `Ask`, `Clipboard`, `CurrentDate`, `ExtensionInput`, `RepeatIndex`, `RepeatItem`). The comment ("originates from `values.py` `MagicVar.to_token()["Type"]` values") is slightly misleading — the set is hardcoded, not imported — but the values are correct.

### 3. `output-uuid-orphan` — PASS

The critical ordering is correct: each action's UUID is added to `seen_uuids` **after** checking its outbound references (line 389: `own_uuid = params.get("UUID")` registered after the inner loop). This means both forward references (action 0 referencing action 1's UUID) and self-referential references are correctly flagged. Simulated trace confirms: when action 0 references UUID `B` and action 1 owns UUID `B`, the reference is caught because `B` is not yet in `seen_uuids`.

The brief asked whether "forward references / self-references flagged as orphan" was validated by a test. Technically no: `test_validate_output_uuid_orphan_flagged` tests a UUID that is absent from the workflow entirely (not a forward reference to a later action). The valid back-reference test (`test_validate_output_uuid_valid_back_ref_is_clean`) exercises the happy path via the schema builder. There is no explicit test for "reference to UUID of a later action" as a distinct case. The logic is correct, but the test gap means a future regression could go unnoticed.

`RawAction` compatibility: `params = action.get("WFWorkflowActionParameters") or {}` at line 348 means actions with no parameters dict (raw plist actions that never carried a UUID) degrade gracefully — no crash, no spurious finding.

### 4. `import-question-action-index-out-of-range` — PASS

The check (`not (0 <= action_index < n_actions)`, line 429) is correct. The missing-`ActionIndex` case (returns `None`) is also handled (line 421), emitting a separate message. The off-by-one test (`ActionIndex == len(actions)`) passes. The valid-index test passes. Straightforward and complete.

## Issues (severity-sorted)

### Medium — Variable-not-set does not catch use-before-set in flat flows

`_collect_set_variable_names` pre-collects all bound names before the reference walk begins. A workflow that uses `MyVar` at action 0 and sets it at action 2 produces no finding. This is a semantic mismatch with the brief ("build the name set as it goes") and with what iOS Shortcuts actually enforces — references to variables not yet set at the point of execution resolve to empty string. The fix is a two-pass walk that accumulates `bound_names` incrementally per action rather than pre-collecting. No test covers this case, so the gap is also unverified. Flag for V1.5 backlog.

### Low — `validate_workflow` not exported from `shortcut_lib/__init__.py`

`from shortcut_lib import validate_workflow` raises `ImportError`. Callers (MCP server, audit CLI) must use `from shortcut_lib.validate import validate_workflow` directly. The brief says "consumers call this function" — that is still possible via direct module import, but it is not part of the public package surface. Whether this belongs in `__all__` is a product decision; flag it for the author.

### Low — `validate.py` is exactly 504 lines (4 over the project 500-line ceiling)

Global instruction: "Max 500 lines per file. Refactor if exceeded." At 504 lines this is a minor breach. The docstring block alone is ~30 lines; trimming or splitting `_validate_envelope` into a helper would bring it under. Not a blocker but should be corrected before merge per house rules.

### Low — `_MAGIC_VAR_TYPES` is hardcoded rather than imported from `values.py`

The comment says the set "originates from `values.py` MagicVar.to_token()['Type'] values" but the values are duplicated as a frozen literal. If a new magic-var singleton is added to `values.py`, `validate.py` silently falls out of sync. The correct pattern is:

```python
from shortcut_lib.schema.values import CurrentDate, Clipboard, Ask, ShortcutInput, RepeatItem, RepeatIndex

_MAGIC_VAR_TYPES: frozenset[str] = frozenset(
    mv.type_name for mv in (CurrentDate, Clipboard, Ask, ShortcutInput, RepeatItem, RepeatIndex)
)
```

The current values match, so this is not a live bug, but the coupling is fragile.

### Info — No explicit forward-UUID-reference test

The orphan test exercises "UUID not present anywhere" but not "UUID present only in a later action." The logic is correct (confirmed by trace), but the test gap means the specific forward-reference claim is unverified by the test suite. A one-line addition to the test file would close this.

### Info — `unknown-envelope` `all_observed_types` recomputed per slot miss

Minor inefficiency: the set of all observed envelope types across the oracle is rebuilt inside the inner loop on every novel-slot miss (line 225). Cache it at the top of `_validate_envelope` as a local variable. Zero user impact at current corpus size (214 oracle slots).
